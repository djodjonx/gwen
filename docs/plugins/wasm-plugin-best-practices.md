> This document captures the concrete rules, pitfalls, and design decisions
> learned while building `@gwen/plugin-physics2d` — the first official GWEN
> WASM plugin. Every section describes **why** a rule exists, not just what it is.
>
> Keep this document up to date as the ecosystem grows. Each new plugin is an
> opportunity to add hard-won knowledge here.

## 1. Project structure rules

### Rule: One crate + one npm package per plugin

Each WASM plugin lives in two places:

```
crates/gwen-plugin-{name}/    ← Rust simulation logic
packages/@gwen/plugin-{name}/ ← TypeScript glue + public API
```

Never mix simulation logic into the TypeScript package or vice versa. The
separation is load-bearing: the Rust crate compiles to a `.wasm` binary
published inside the npm package's `wasm/` directory.

### Rule: `wasm/` is always gitignored

```gitignore
# packages/@gwen/plugin-{name}/wasm/.gitignore
*
!.gitignore
```

WASM artefacts (`*.js`, `*.wasm`, `*.d.ts`) are generated by `wasm-pack` and
**never committed**. They are published to npm as part of the package build.
Committing them creates merge conflicts and a false sense of security (the
committed binary may be outdated).

### Rule: Always declare `gwen.wasmFiles` in `package.json`

```json
{
  "gwen": {
    "type": "wasm-plugin",
    "wasmId": "physics2d",
    "wasmFiles": ["wasm/gwen_physics2d.js", "wasm/gwen_physics2d_bg.wasm"]
  }
}
```

`@gwen/vite-plugin` scans `node_modules/@gwen/plugin-*/wasm/` to serve files
in dev and copy them to `dist/wasm/` for production. Without this field, the
vite-plugin cannot know which files to serve.

### Rule: Follow the 4-file Rust structure

Every GWEN WASM plugin crate has exactly four source files:

| File | Responsibility |
|------|----------------|
| `src/lib.rs` | Re-exports only. No logic. |
| `src/bindings.rs` | All `#[wasm_bindgen]` exports. The WASM boundary. |
| `src/world.rs` | Simulation state and logic. Pure Rust, no WASM concern. |
| `src/memory.rs` | Read/write helpers for the shared buffer. Copy from `gwen-plugin-physics2d`. |

This separation ensures that `world.rs` is testable in pure Rust (no browser,
no WASM) and that `bindings.rs` stays thin (just delegation + type adaptation).

## 2. Rust crate rules

### Rule: Always use `crate-type = ["cdylib", "rlib"]`

```toml
[lib]
crate-type = ["cdylib", "rlib"]
```

- `cdylib` — required for `wasm-pack` to produce a `.wasm` binary.
- `rlib` — required for `cargo test` to run unit tests natively (no browser).

Without `rlib`, `cargo test -p gwen-plugin-{name}` fails to compile.

### Rule: Disable `wasm-opt` in release profiles

```toml
[package.metadata.wasm-pack.profile.release]
wasm-opt = false
```

`wasm-opt` (from Binaryen) can trip on bulk-memory or SIMD opcodes introduced
by Rapier or other simulation libraries. Disabling it avoids subtle correctness
bugs with no measurable runtime cost for game-scale binaries (< 2 MB).

### Rule: Accept `shared_ptr: usize` as the first constructor argument

```rust
#[wasm_bindgen(constructor)]
pub fn new(shared_ptr: usize, max_entities: u32) -> Self { … }
```

The TypeScript layer passes `region.ptr` (a raw address in gwen-core's linear
memory) as the first argument. Using `usize` — not `u32` or `i32` — ensures
correctness on both 32-bit and 64-bit WASM targets.

### Rule: All pointer arithmetic is `unsafe` — keep it in `memory.rs`

Never scatter raw pointer operations across `world.rs` or `bindings.rs`. All
unsafe memory access must go through the typed helpers in `memory.rs`:

```rust
// ✅ Correct — through the helper
let (x, y, rot) = unsafe { memory::read_position_rotation(shared_ptr, entity_index) };

// ❌ Wrong — naked pointer arithmetic in world.rs
let x = unsafe { *((shared_ptr + entity_index as usize * 32) as *const f32) };
```

This rule makes auditing unsafe code trivial: one file, one responsibility.

### Rule: Use `#[wasm_bindgen]` only on the public contract

Only mark structs and methods that TypeScript actually calls with
`#[wasm_bindgen]`. Internal helpers, state structs, and free functions used
only from Rust should not carry the attribute — it increases binary size and
the risk of accidental exposure.

### Rule: Return JSON strings for complex results across the WASM boundary

wasm-bindgen supports returning `String` efficiently. For anything more complex
than a scalar (e.g. collision events, stats, position tuples), return a JSON
string and parse it in TypeScript:

```rust
// Rust
pub fn get_collision_events(&self) -> String {
    // Build JSON manually — no serde overhead for a list of { a, b, started }
    let mut s = String::from('[');
    for (i, ev) in self.events.iter().enumerate() {
        if i > 0 { s.push(','); }
        s.push_str(&format!(r#"{{"a":{},"b":{},"started":{}}}"#,
            ev.entity_a, ev.entity_b, ev.started));
    }
    s.push(']');
    s
}
```

```typescript
// TypeScript
const events: CollisionEvent[] = JSON.parse(wasmPlugin.get_collision_events());
```

Avoid returning `Vec<JsValue>` — it allocates a JS Array on every call, which
triggers the garbage collector on the hot path.

### Rule: Respect `MemoryRegion.byteLength` — never write beyond it

`SharedMemoryManager` places a `0xDEADBEEF` sentinel immediately after your
region. In debug mode (`engine: { debug: true }`), `checkSentinels()` is called
every frame. An overrun causes an immediate, named error:

```
[GWEN:SharedMemory] Sentinel overwrite detected for plugin 'my-plugin'!
```

To avoid this: always compute your maximum write offset before the first frame.

```rust
// In new(): verify the allocation fits
let max_write_offset = max_entities as usize * TRANSFORM_STRIDE;
assert!(
    max_write_offset <= /* your declared sharedMemoryBytes */,
    "Plugin would write past its MemoryRegion"
);
```

## 3. Shared memory rules

### Rule: `sharedMemoryBytes = maxEntities × TRANSFORM_STRIDE`

```typescript
readonly sharedMemoryBytes: number;

constructor(config: MyPluginConfig = {}) {
  this.sharedMemoryBytes = (config.maxEntities ?? 10_000) * TRANSFORM_STRIDE;
}
```

`TRANSFORM_STRIDE` is always 32 bytes. Import it from `@gwen/engine-core`.
**Never hard-code 32** — if the stride ever changes, your plugin will silently
read corrupt data.

### Rule: Declare `sharedMemoryBytes` statically

`SharedMemoryManager.allocateRegion()` is called **before** `onInit()`. The
engine must know how much memory to reserve at startup. This value must be
deterministic and not depend on runtime state.

### Rule: `maxEntities` must match the engine's `maxEntities`

If your plugin uses entity_index as a direct array offset into the shared
buffer, entity indices above your `maxEntities` will write out of bounds.
The safest pattern: accept `maxEntities` in your config and default it to
the same value as the engine:

```typescript
export default defineConfig({
  engine: { maxEntities: 10_000 },
  wasmPlugins: [myPlugin({ maxEntities: 10_000 })], // must match
});
```

A future version of GWEN may inject `maxEntities` automatically via `onInit`.

### Rule: Never cache `memory.buffer` across frames

`WebAssembly.Memory.buffer` is replaced when Rust calls `memory.grow()`.
Any `TypedArray` or `DataView` built on the old buffer becomes silently
detached — reads return 0, writes are discarded.

```typescript
// ❌ Wrong: buffer cached in a field
this.view = new Float32Array(bridge.getLinearMemory()!.buffer, ptr, n);

// ✅ Correct option A: re-wrap every frame
const view = new Float32Array(bridge.getLinearMemory()!.buffer, ptr, n);

// ✅ Correct option B: implement onMemoryGrow() to refresh on demand
onMemoryGrow(newMemory: WebAssembly.Memory) {
  this.view = new Float32Array(newMemory.buffer, this.ptr, this.n);
}
```

In practice, most plugins **never need a TypeScript view** over the shared
buffer — the Rust plugin reads and writes directly without JS involvement.
The rule matters mainly for debug-overlay plugins that visualise the buffer.

## 4. TypeScript glue layer rules

### Rule: Always implement `_prefetch()` for production plugins

```typescript
private _wasmModule: MyWasmModule | null = null;

async _prefetch(): Promise<void> {
  this._wasmModule = await loadWasmPlugin<MyWasmModule>({
    jsUrl:   '/wasm/gwen_my_plugin.js',
    wasmUrl: '/wasm/gwen_my_plugin_bg.wasm',
    name:    this.name,
  });
}

async onInit(bridge, region, api) {
  // Module is already loaded — no second network round-trip
  const wasm = this._wasmModule ?? await loadWasmPlugin(…);
  this.wasm = new wasm.MyPlugin(region.ptr, this.maxEntities);
}
```

`createEngine()` runs all `_prefetch()` calls in parallel before the sequential
`onInit()` phase. Without `_prefetch()`, each plugin's `.wasm` binary is fetched
one after the other — O(n) load time instead of O(1).

### Rule: Use `loadWasmPlugin()` — never raw `import()` or `fetch()`

```typescript
import { loadWasmPlugin } from '@gwen/engine-core';

const wasm = await loadWasmPlugin<MyWasmModule>({
  jsUrl:   '/wasm/gwen_my_plugin.js',
  wasmUrl: '/wasm/gwen_my_plugin_bg.wasm',
  name:    'MyPlugin',
});
```

`loadWasmPlugin()` uses a blob-script-tag strategy that bypasses Vite's
`import()` restrictions on `/public` assets, caches the module by URL
(idempotent), and handles both `default()` (async) and `initSync()` init forms.
Rolling your own fetch breaks in Vite dev mode and misses the caching.

### Rule: `onInit()` is for init only — no simulation logic

`onInit()` must:
1. Load the WASM module (or use the `_prefetch` cache).
2. Instantiate the Rust struct with `region.ptr`.
3. Register the service via `api.services.register()`.

It must **not** run game logic, start loops, or schedule tasks. Those belong
in `onStep()` and `onUpdate()`.

### Rule: Call `api.services.register()` inside `onInit()`, never in `provides`

The `provides` field is a **type-declaration hint** for TypeScript inference —
its runtime values are never read. The actual registration happens in `onInit()`:

```typescript
// ✅ Correct
async onInit(bridge, region, api) {
  api.services.register('physics', this._createAPI());
}

// ❌ Wrong — service never reaches the ServiceLocator
readonly provides = { physics: this._createAPI() }; // _createAPI() too early
```

### Rule: Guard `onStep()` against pre-init calls

```typescript
onStep(deltaTime: number): void {
  // wasmPlugin is null before onInit() completes — graceful no-op
  this.wasmPlugin?.step(deltaTime);
}
```

The engine guarantees `onInit()` finishes before `start()` is called, but
defensive coding prevents crashes during hot-reload or test setup.

### Rule: `onDestroy()` must free the Rust instance

wasm-bindgen-generated structs have a `free()` method. Not calling it leaks
the Rust heap allocation for the lifetime of the page:

```typescript
onDestroy(): void {
  this.wasmPlugin?.free?.();
  this.wasmPlugin = null;
}
```

Setting `wasmPlugin = null` ensures subsequent `onStep()` calls are no-ops.

### Rule: Expose a typed service API — never expose the raw wasm instance

```typescript
// ❌ Wrong: leaks the raw Rust binding into game code
api.services.register('physics', this.wasmPlugin);

// ✅ Correct: clean TypeScript API that hides the Rust object
api.services.register('physics', this._createAPI());
```

The raw wasm instance uses `snake_case` method names, has no documentation,
and exposes Rust internals. The service API is the plugin's public contract.

## 5. Lifecycle rules

### The 8-step game loop (do not reorder)

```
1. dispatchBeforeUpdate()    TsPlugins: capture inputs, intentions
2. syncTransformsToBuffer()  ECS → shared buffer (before Rust reads)
3. dispatchWasmStep()        Rust simulation step
4. checkSentinels()          [debug only] verify no buffer overruns
5. syncTransformsFromBuffer() shared buffer → ECS (after Rust writes)
6. wasmBridge.tick()         Rust game-loop heartbeat
7. dispatchUpdate()          TsPlugins: game logic on updated values
8. dispatchRender()          TsPlugins: rendering
```

Your plugin's `onStep()` is called at step 3. This means:
- You can read ECS transform data (written in step 2).
- TypeScript game logic runs *after* you (step 7) — your results are available.
- Never call `api.services.get()` inside `onStep()` — use `onInit()` to cache
  the reference.

### Rule: Resolve services in `onInit()`, use them in `onStep()`/`onUpdate()`

```typescript
private physicsAPI!: Physics2DAPI;

onInit(api: EngineAPI) {
  // One-time resolution — O(1) hash map lookup
  this.physicsAPI = api.services.get('physics') as Physics2DAPI;
}

onUpdate(api: EngineAPI) {
  // Use the cached reference — zero lookup cost
  this.physicsAPI.applyImpulse(entityIndex, 0, 500);
}
```

Service resolution is not expensive, but the pattern makes dependencies
explicit and enables easy testing (inject the service in tests).

### Rule: `createEngine()` must be `await`-ed

```typescript
// ✅ Correct
const { engine, scenes } = await createEngine(gwenConfig, registerScenes, mainScene);

// ❌ Wrong — WASM plugins not initialized, engine.start() will crash
const { engine } = createEngine(gwenConfig);
```

`createEngine()` became async when WASM plugin support was added. It awaits
`_prefetch()` + `onInit()` for every declared `wasmPlugin` before returning.

## 6. Service API design rules

### Rule: Type your service API as an interface, not a class

```typescript
// types.ts
export interface Physics2DAPI {
  addRigidBody(entityIndex: number, type: RigidBodyType, x: number, y: number): number;
  applyImpulse(entityIndex: number, x: number, y: number): void;
  getCollisionEvents(): CollisionEvent[];
  getPosition(entityIndex: number): { x: number; y: number; rotation: number } | null;
}
```

An interface:
- Can be mocked trivially in tests.
- Makes the contract explicit and independent of the implementation.
- Allows a future pure-TypeScript fallback implementation.

### Rule: Use numeric enums for types that cross the WASM boundary

```typescript
// types.ts
export const BODY_TYPE: Record<RigidBodyType, number> = {
  fixed:     0,
  dynamic:   1,
  kinematic: 2,
} as const;
```

```typescript
// index.ts — in _createAPI()
addRigidBody(entityIndex, type, x, y) {
  return self.wasmPlugin!.add_rigid_body(entityIndex, x, y, BODY_TYPE[type]);
}
```

Passing a numeric discriminant across the WASM boundary is zero-cost.
Passing strings forces Rust to do string comparison — avoid it.

### Rule: Return `null` for "not found", never throw

```typescript
getPosition(entityIndex: number): { x: number; y: number; rotation: number } | null {
  const result = self.wasmPlugin?.get_position(entityIndex);
  if (!result || result.length < 3) return null;
  return { x: result[0], y: result[1], rotation: result[2] };
}
```

Throwing from a service API inside `onUpdate()` causes an uncaught exception
that stops the game loop. Returning `null` lets the caller decide how to handle
the missing case.

### Rule: `getCollisionEvents()` returns a snapshot — call it once per frame

Collision events are accumulated by the Rust simulation during `onStep()` and
flushed at the end of the step. Calling `getCollisionEvents()` multiple times
per frame is wasteful (JSON parse on every call). Cache the result:

```typescript
onUpdate(api: EngineAPI) {
  // ✅ One parse per frame
  const events = this.physics.getCollisionEvents();
  for (const ev of events) { /* … */ }
}
```

## 7. Testing rules

### Rule: Test Rust world logic with native tests, not browser tests

```bash
# Fast: runs in milliseconds, no browser, no WASM compilation
cargo test -p gwen-plugin-my-plugin
```

The key insight: `world.rs` contains pure Rust — it can be tested natively with
`cargo test` without compiling to WASM. Browser tests (`wasm-pack test --chrome`)
are 10-50x slower and should be reserved for testing the WASM boundary itself.

### Rule: Test the TypeScript glue with mocked WASM

Never load a real `.wasm` file in TypeScript unit tests. Mock `loadWasmPlugin`
and return a plain object with `vi.fn()` methods:

```typescript
vi.mock('@gwen/engine-core', async (importOriginal) => {
  const original = await importOriginal<typeof import('@gwen/engine-core')>();
  return { ...original, loadWasmPlugin: vi.fn() };
});

const mockWasm = {
  MyPlugin: vi.fn().mockReturnValue({
    step: vi.fn(),
    get_position: vi.fn().mockReturnValue([1.0, 2.0, 0.5]),
    free: vi.fn(),
  }),
};
(loadWasmPlugin as Mock).mockResolvedValue(mockWasm);
```

This pattern:
- Runs in Node.js (no browser).
- Completes in milliseconds.
- Tests TypeScript logic in isolation from Rust.
- Verifies that the correct arguments reach the WASM boundary.

### Rule: Test at least these 8 scenarios for every plugin

| # | Scenario | Why |
|---|----------|-----|
| 1 | `id` and `name` are correct constants | Typos break service lookup |
| 2 | `sharedMemoryBytes = maxEntities × 32` | Wrong value causes OOB writes |
| 3 | `onInit()` calls `loadWasmPlugin` with the correct URLs | URL mismatch = silent 404 |
| 4 | `onInit()` passes `region.ptr` to the Rust constructor | Wrong ptr = wrong memory |
| 5 | `onInit()` registers the service | Missing registration = crash in onUpdate |
| 6 | `onStep()` delegates to `wasm.step()` | Simulation never runs |
| 7 | `onStep()` is safe before `onInit()` | Prevents crash during hot-reload |
| 8 | `onDestroy()` calls `wasm.free()` | Memory leak on stop/restart |

### Rule: Use `makeMockBridge()` from `shared-memory.test.ts` as a template

The mock bridge in `packages/@gwen/engine-core/tests/shared-memory.test.ts`
includes a real in-process `ArrayBuffer` as fake WASM memory — it lets you
test sentinel reads/writes without a browser. Copy this pattern for plugins
that need to verify memory layout assumptions.

## 8. Performance rules

### Rule: Iterate over entities in Rust, not TypeScript

```rust
// ✅ Correct — tight loop in Rust, no JS overhead
pub fn step(&mut self, delta: f32) {
    for entity_index in 0..self.max_entities {
        if self.active[entity_index as usize] {
            // Compute and write directly to shared buffer
        }
    }
}
```

```typescript
// ❌ Wrong — 10,000 WASM boundary crossings per frame
for (let i = 0; i < maxEntities; i++) {
  if (physics.isActive(i)) physics.stepEntity(i, delta);
}
```

Each JavaScript → WASM function call has a small but non-zero overhead. A tight
loop over 10,000 entities with one call per entity adds up to ~1–5 ms per frame.
The same loop in Rust completes in microseconds.

### Rule: Batch all per-entity reads and writes through the shared buffer

The shared buffer is the zero-copy data channel. Design your Rust step function
to read all inputs, compute all outputs, and write all results in a single pass —
not through individual getter/setter calls across the WASM boundary:

```rust
pub fn step(&mut self, delta: f32) {
    // One pass: read input → simulate → write output
    for i in 0..self.max_entities {
        let (x, y, rot) = unsafe { memory::read_position_rotation(self.ptr, i) };
        let (nx, ny, nrot) = self.world.simulate(i, x, y, rot, delta);
        unsafe { memory::write_position_rotation(self.ptr, i, nx, ny, nrot); }
    }
}
```

### Rule: JSON serialization is for infrequent data only

JSON (`get_collision_events()`, `stats()`) is acceptable for data that changes
rarely or has bounded size. Do **not** use JSON for per-entity position reads —
that is what the shared buffer is for.

## 9. Common pitfalls

### Pitfall: `createEngine()` not awaited

**Symptom:** `api.services.get('physics')` returns `undefined`.
**Cause:** `createEngine()` returns before `onInit()` has finished.
**Fix:** `const { engine } = await createEngine(…)`.

---

### Pitfall: Wrong `sharedMemoryBytes`

**Symptom:** Sentinel overwrite error in debug mode, or silent data corruption
in production.
**Cause:** `sharedMemoryBytes` is smaller than the actual bytes written by Rust.
**Fix:** `sharedMemoryBytes = maxEntities * TRANSFORM_STRIDE`. Enable `debug: true`
in development — sentinel guards catch this immediately.

---

### Pitfall: `maxEntities` mismatch between engine and plugin

**Symptom:** Entities with index > plugin's `maxEntities` are invisible to the
physics simulation even though they exist in the ECS.
**Cause:** Plugin was created with `maxEntities: 1000` while engine uses `10_000`.
**Fix:** Always pass the same `maxEntities` to both:

```typescript
const MAX = 10_000;
defineConfig({
  engine: { maxEntities: MAX },
  wasmPlugins: [physics2D({ maxEntities: MAX })],
});
```

---

### Pitfall: COOP/COEP headers missing in production

**Symptom:** Works in `vite dev` (headers injected automatically), crashes in
production (`crossOriginIsolated = false`).
**Cause:** Production server does not send the isolation headers.
**Fix:** Configure your server:

```nginx
add_header Cross-Origin-Opener-Policy  "same-origin";
add_header Cross-Origin-Embedder-Policy "require-corp";
```

---

### Pitfall: Caching `memory.buffer` across frames

**Symptom:** All reads return 0 or stale values after a few seconds of gameplay.
**Cause:** Rust triggered a `memory.grow()` and the cached `ArrayBuffer` was
detached. The old buffer is kept alive by the reference but all operations are
no-ops.
**Fix:** Re-wrap `memory.buffer` on every use, or implement `onMemoryGrow()`.

---

### Pitfall: Registering the service outside `onInit()`

**Symptom:** TypeScript plugins get `undefined` from `api.services.get('myService')`.
**Cause:** `api.services.register()` was called in the constructor or in `provides`.
**Fix:** Always register inside `onInit()`:

```typescript
async onInit(bridge, region, api) {
  // …
  api.services.register('myService', this._createAPI()); // ← here only
}
```

---

### Pitfall: `free()` not called on `onDestroy()`

**Symptom:** Memory leak visible in DevTools after repeatedly starting/stopping
the engine (e.g. scene transitions that recreate the engine).
**Cause:** `this.wasmPlugin.free()` was omitted.
**Fix:**

```typescript
onDestroy(): void {
  this.wasmPlugin?.free?.();
  this.wasmPlugin = null;
}
```

---

### Pitfall: `add_rigid_body()` on a body without a collider (Rapier-specific)

**Symptom:** `applyImpulse()` has no effect. Body position never changes.
**Cause:** In Rapier, a dynamic body without a collider has zero mass and is
immediately put to sleep. The simulation ignores it.
**Fix:** Always add at least one collider after creating a body:

```typescript
const handle = physics.addRigidBody(entityIndex, 'dynamic', x, y);
physics.addBallCollider(handle, 0.5); // required for mass computation
```

## 10. Decision log

This section records the non-obvious architectural choices made during
`gwen-plugin-physics2d` development, with the reasoning behind each decision.

---

### D1: Pointer-based shared buffer instead of `SharedArrayBuffer`

**Decision:** The shared buffer lives in `gwen-core`'s WASM linear memory,
accessed via a raw `usize` pointer — not a JS `SharedArrayBuffer`.

**Rationale:**
- `SharedArrayBuffer` requires COOP/COEP headers on the **host HTML page**,
  which breaks loading of third-party resources (CDN images, fonts, audio).
  Many production games cannot afford this constraint.
- `gwen-core`'s WASM memory is already accessible to all WASM plugins via the
  `ptr` convention — no extra browser feature flag needed.
- Performance is equivalent: both approaches avoid copying data through JS.

**Trade-off:** Multi-threaded WASM (worker threads sharing the buffer) is not
possible with this approach. Acceptable for v1 — GWEN is single-threaded.

---

### D2: Two-phase plugin initialisation (`_prefetch` + `onInit`)

**Decision:** `createEngine()` separates loading into a parallel fetch phase
(`_prefetch`) and a sequential init phase (`onInit`).

**Rationale:**
- Parallel fetch reduces load time from O(n × fetch_time) to O(max_fetch_time).
- Sequential init is required because `allocateRegion()` mutates shared state
  (`usedBytes`) and is not concurrency-safe. A race condition here would cause
  regions to overlap silently.
- The split is clean: `_prefetch` is pure network I/O, `onInit` is memory + DI.

**Alternative considered:** `Promise.all(plugins.map(p => p.onInit()))` —
rejected because it races on `allocateRegion()`.

---

### D3: `0xDEADBEEF` sentinel guards in debug mode only

**Decision:** Sentinel checks run only when `engine: { debug: true }`.

**Rationale:**
- In production, sentinel checks add one `DataView.getUint32()` call per plugin
  per frame. This is negligible (~50 ns total) but the principle of "zero cost
  in prod" is important for user confidence.
- In development, the check is invaluable: a buffer overrun produces a named,
  actionable error instead of silent data corruption that surfaces 10 frames
  later as a physics glitch.

---

### D4: JSON strings for event data across the WASM boundary

**Decision:** `get_collision_events()` returns a `String` containing JSON, not
a `Vec<JsValue>` or a flat `Uint32Array`.

**Rationale:**
- `Vec<JsValue>` allocates a JS Array + individual `JsValue` objects per event
  on every call — this triggers the garbage collector on the hot path.
- A flat `Uint32Array` would be faster but requires a stable binary layout and
  versioning strategy between the Rust and TypeScript sides.
- JSON is human-readable, debuggable, and parsed in one allocation. For a list
  of at most ~100 collision events per frame, the parse cost is negligible.

**Future:** If profiling shows JSON parsing is a bottleneck, replace with a
typed flat buffer (e.g. `[entity_a: u32, entity_b: u32, started: u8, …]`).

---

### D5: `getLinearMemory()` returns `null` in test environments

**Decision:** `_injectMockWasmEngine()` leaves `_wasmModule = null`, so
`getLinearMemory()` returns `null`. `checkSentinels()` and `_writeSentinels()`
are no-ops when memory is null.

**Rationale:**
- Test environments run in Node.js, which has no `WebAssembly.Memory`.
- Requiring tests to construct a fake `WebAssembly.Memory` is unnecessary
  boilerplate for tests that don't exercise the memory layout.
- The null guard is explicit: sentinel checks are not skipped silently in prod
  (where `_wasmModule` is always set by `initWasm()`).
